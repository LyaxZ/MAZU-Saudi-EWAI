## 一、项目概述与分工（1 分钟）🕐 0:00-1:00

各位老师好。我们的项目是 **MAZU 沙特多灾种预警智能体**——用 LightGBM + 知识图谱 + LLM Agent 三层混合架构，为沙特阿拉伯构建「预测→解释→推理→行动」闭环的多灾种预警原型。

### 工作展开方式

我们按照「数据→模型→知识→Agent」的 Pipeline 思路逐步推进：

1. **数据层**：从 MAZU 系统的 91 个气象指标出发，处理 365 天 × 35,200 格点 = 1,284 万条样本

3. **知识层**：构建双层知识图谱（概念因果图 + 物理 D8 流向图），实现空间风险推理
4. **Agent 层**：基于 DeepSeek Function Calling，将模型、图谱、处置框架集成为自然语言交互系统

## 二、数据底座与工程挑战（1 分钟）🕐 1:00-2:00

### 技术方法：MAZU 数据 Pipeline

MAZU 系统从 11 种全球气象产品（GRIB2/NetCDF/HDF5/地面站点）中裁剪沙特区域（16-32°N, 34-56°E），计算 91 个指标变量，涵盖大气廓线、地表热力、水汽、风场、降水五大类。覆盖 2025 年全年 365 天、160×220 网格（0.1° × 0.1°，约 11 km），合计约 **1,284 万条样本**。

### 数据工程中的关键挑战与解决方法

| 挑战 | 问题描述 | 解决方法 |
|---|---|---|
| NC 文件变量不均 | 1 月缺 26 个廓线变量，12 月缺 9 个卫星降水 | `fillna(0)` 保留样本，空间 CV 验证稳健性（CSI 均值 0.932） |
| SST 坐标不兼容 | `(lat,lon)` vs `(latitude,longitude)` 三处差异 | xarray nearest 插值对齐，不生成新值 |
| 标签由物理规则生成 | 非真实灾害记录，由 CAPE+降水+湿度联合判定 | 四层验证体系（规则拟合+物理一致+真实事件+随机基线） |

这三个挑战的解决贯穿了整个项目，后续各节会逐一展开具体方案。

## 三、系统架构：三层混合范式（1.5 分钟）🕐 2:00-3:30

我们的核心设计理念：**不同维度的问题用不同范式的工具解决**，而非追求端到端的单一模型。

```
用户自然语言
    ↓
┌─────────────────────────────────────┐
│  顶层：LLM Agent (DeepSeek API)     │  回答「该怎么做？」
│  Function Calling 调度三工具        │  意图解析 → 工具编排 → 综合回复
└──────────────┬──────────────────────┘
               ↓ 风险概率 + SHAP 解释
┌─────────────────────────────────────┐
│  中层：NetworkX 知识图谱             │  回答「影响范围多大？」
│  35,200 节点 + 279,324 边           │  D8 流向 / 风场扇形 / 沿海增水
│  RiskPropagator 四种传播策略        │
└──────────────┬──────────────────────┘
               ↓ 35,200 格点 × 25 特征
┌─────────────────────────────────────┐
│  底层：LightGBM 预测引擎             │  回答「哪里有多高风险？」
│  四灾害独立建模                     │  2 分钟训练 / 324 万样本
│  15-25 维物理特征 + 经纬度编码      │
└─────────────────────────────────────┘
```

### 底层 — LightGBM 预测引擎

四灾害独立建模，6-8 月训练（92 天 × 35,200 = 324 万样本），9 月验证，10 月测试。特征涵盖六大类物理量：降水（日总量、对流降水比）、对流（CAPE/CIN/风切变）、水汽（相对湿度/水汽通量散度）、风场（10m 风速/850hPa 涡度）、热力（气温距平/VPD/地表温度）、地形（orography 高度）。特征数 15-25 维。单模型训练约 2 分钟，体积 ~5MB，适合生产部署。

### 中层 — NetworkX 知识图谱

全图 35,200 节点。边分两类：（1）空间邻接边 245,482 条；（2）D8 地形流向边 33,842 条。四种差异化传播策略：

| 灾害 | 策略 | 机制 | 参数 |
|---|---|---|---|
| 山洪 | `downstream_flow` | 沿 D8 流向 BFS，指数衰减 | β=0.05/km |
| 沙尘 | `wind_fan` | 按 10m 风向 ±30° 扇形扩散 | — |
| 高温 | `exposure_summary` | 网格内人口/设施暴露度汇总 | — |
| 风浪 | `coastal_inland` | 海岸线向内陆 ≤30km | — |

### 顶层 — LLM Agent

基于 DeepSeek API + Function Calling。三个工具：`predict_risk`（风险预测 + SHAP 特征贡献）、`query_kg_impact`（影响链 + 承灾体查询）、`search_similar_cases`（15 条真实灾害种子案例检索）。

## 四、模型选型：LightGBM 为什么比 LSTM 好？（1.5 分钟）🕐 3:30-5:00

### 技术方法对比

我们最初假设气象数据的时空连续性适合 LSTM。为此用 PyTorch 构建了 3 层 BiLSTM（隐藏维度 256，7 天滑动窗口），训练了规模梯度：

| 样本量 | CSI | POD | FAR | AUC | 训练时间 |
|---|---|---|---|---|---|
| 2 万 | 0.501 | 0.949 | 0.485 | 0.922 | ~5 min |
| 10 万 | 0.629 | 0.939 | 0.345 | 0.981 | ~20 min |

增加样本量 CSI 从 0.501 提升到 0.629，但收益递减，趋近于上限。而 LightGBM 在全量 324 万样本上 CSI 达到 1.000，训练仅需 2 分钟。

### 原因分析

差距不在于模型优劣，而在于**任务形态与模型归纳偏置的匹配度**：

- 输入：单日多维表格特征（91 个物理指标 × 1 个时间点）
- 标签：同日是否发生灾害（二分类）
- 本质：**结构化表格分类**，不是时序预测

LSTM 的门控机制旨在捕捉长序列时间依赖，但我们的标签构建已包含时序信息（如 `heatwave_day_flag` 编码了连续高温天数），表格特征已足够表达。LSTM 的时序建模能力在此场景下是冗余的，反而引入梯度消失、训练不稳定等问题。

我们还尝试了 XGBoost 联合建模、衍生特征、空间平滑，均无增益——说明单 LightGBM 已逼近数据的信息上限。

### 方法论启示

**选对工具比用复杂工具更重要。** 极端高温从 CSI 0.283→0.997 靠的是特征工程，不是更复杂的模型；SST 从无法入模到第二重要特征，靠的是坐标对齐，不是更强的算法。

## 五、三项关键优化（2.5 分钟）🕐 5:00-7:30

### 一、经纬度周期编码（40 秒）

直接将经纬度作为数值输入，模型无法理解球面周期性。我们为每个格点添加周期编码：

$$lat_{sin} = \sin(2\pi \cdot lat / 180),\quad lat_{cos} = \cos(2\pi \cdot lat / 180)$$
$$lon_{sin} = \sin(2\pi \cdot lon / 360),\quad lon_{cos} = \cos(2\pi \cdot lon / 360)$$

效果：CSI 提升 0.009，FAR 下降 31%。隐含收益：相邻格点编码相似，模型自动学到空间聚类。

### 二、极端高温特征工程（1 分钟）

极端高温优化是一个完整的三阶段案例：

| 阶段 | 手段 | CSI | 关键认知 |
|---|---|---|---|
| 初始 | 基础特征 | 0.283 | 模型无法区分"夏季正常高温"与"异常热浪" |
| 第一阶段 | + 经纬度编码 + 阈值优化(0.50→0.75) | 0.618 | 空间信息有用，但核心问题未解决 |
| 第二阶段 | + 气温距平 + 气候态均值 | **0.997** | **异常信号比绝对值更具预测价值** |

核心突破：引入 `tmax_anomaly_c`（当日偏离气候态的程度）和 `tmax_climatology_c`（历史均值），模型获得参照系。例如：6 月利雅得 45°C 正常，但 1 月 30°C 就是极端——没有气候态参照无法判断。

SHAP 验证：气温距平 Gain 贡献占 70.3%，确认了异常信号的主导地位。

### 三、SST 海温坐标对齐（50 秒）

`sea_surface_temperature` 使用独立网格，与主网格存在三处不兼容：

| 差异 | SST 网格 | 主网格 |
|---|---|---|
| 纬度命名/方向 | `lat`，升序（16→32） | `latitude`，降序（32→16） |
| 偏移 | ~0.05° 偏差 | — |
| 经度点数 | 221 | 220 |

解决方法：xarray `interp(method='nearest')` → 裁剪多余经度 → 翻转纬度。关键原则：**不生成新值，仅坐标重映射**。结果：SST 入选 coastal_wave 第二重要特征（Gain 287 万），仅次于 `wind10_speed`，物理机理与海气交换一致。

> 💡 该优化在 GPU 笔记本（RTX 4060, CUDA 12.1）上完成验证。CPU 训练含 SST 的模型约 7 分钟/模型。

## 七、SHAP 可解释性（1 分钟）🕐 9:00-10:00

预警系统不能是黑箱。我们集成 SHAP TreeExplainer，每次预测自动输出 Top-5 特征贡献与自然语言摘要。

| 事件 | Top-1 特征 | 贡献 | 物理含义 |
|---|---|---|---|
| 8/28 阿西尔山洪 | CAPE | 89.4% | 对流有效位能→深对流→暴雨 |
| 5/25 52.2°C 高温 | 气温距平 | 70.3% | 偏离气候态程度→热浪强度 |
| 8/14 塔伊夫冰雹洪水 | CAPE | 85.7% | 强对流+地形抬升→冰雹+山洪 |

三个案例的 SHAP 解释均与气象机理高度吻合——模型学到的是有物理意义的因果信号，而非数据中的虚假相关性。

Agent 生成回复时，将 SHAP 贡献自然融入背景分析：「本次高风险主要由对流有效位能 CAPE 异常偏高（贡献 89%）驱动，配合地形抬升作用，阿西尔山区极易触发强对流降水。」

## 十一、总结（1 分钟）🕐 14:00-15:00

### 工作回顾：怎么做的

我们按照「数据→模型→知识→Agent」Pipeline 系统推进：先处理 MAZU 91 指标 1,284 万样本的数据工程挑战，再通过 LightGBM vs LSTM 对比实验确定技术路线，接着用三项关键优化（周期编码、气温距平、SST 对齐）提升模型性能，然后构建双层 KG 实现空间推理，最后将模型/图谱/处置框架集成为 LLM Agent 自然语言交互系统。

### 技术方法总结

| 层面 | 技术方法 | 解决的问题 |
|---|---|---|
| 预测 | LightGBM 四灾害独立建模 + SHAP | 哪里有多高风险？为什么？ |
| 推理 | NetworkX D8 流向 + 四种传播策略 | 风险如何扩散？影响什么设施？ |
| 行动 | 风险等级×灾害×区域 标准矩阵 | 谁该做什么？ |
| 交互 | DeepSeek Function Calling + Few-Shot | 如何让用户自然语言获取以上信息？ |

### 最终效果

四灾害 LightGBM CSI ≥ 0.99，2025 年 10 次真实灾害事件独立验证 **9/10（90%）** 命中。SHAP 三项案例验证特征贡献与气象机理 100% 吻合。双层 KG 实现从因果解释到物理传播的完整推理链。端到端 Gradio 自然语言对话可演示。
